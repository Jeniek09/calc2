SOLID Object Oriented Design

SOLID is an acronym for the five object-oriented design principles. These principles establish practices that help develop software with factors for maintaining and extending in the project. Incorporating these principles and practices one can avoid refactoring code, code smell and overall organize your code.

S : Single Responsibility Principle
O : Open-Closed Principle
L : Liskov Substitution Principle
I : Interface Segregation Principle
D : Dependency Inversion Principle

Single Responsibility Principle (SRP)

A function should perform one and only one task according to this principle.

Example: when calling static method, if the single responsibility rule is well established then it will return only one action upon calling that class.

Open/Closed Principle (OCP)

Functions and classes should be open for extension and closed modification. Basically, one should be able to add on the function but not go and modify the code each time something gets added.

Example: in the addition calculator test you can have a number (value_a) added to 0 or add a bunch of numbers together (value_a + value_b…).

Liskov Substitution Principle (LSP)

Defines that object of a superclass shall be replaceable with objects of its subclasses without breaking the application.

Example: Example: in the calc folder the calculation.py this calculation class is extended in multiplication or subtraction calculation. Parent class should never depend on child class, but child class won’t be able to execute the result without the parent class.

Interface Segregation Principle (ISP)

The goal is to not include any unwanted methods in an interface. Because declaring methods in an interface that the client doesn’t need pollutes the interface and leads to a “bulky” or “fat” interface, which will affect the readability and maintainability of the code.

Dependency Inversion Principle (DIP)

This principle or rule prevents your code from getting too tied up and makes is easier to reuse. The object of OOP is to make sure your code reusable and decency inversion is one way to achieve that goal
